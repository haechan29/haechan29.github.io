## 인덱스
인덱스(index)는 데이터의 검색 속도를 향상시키기 위해 사용되는 데이터 구조<br/>
대표적으로 B-트리, 해시테이블 등을 사용한다.<br/>

### 인덱스의 장점
- 검색 속도 향상

### 인덱스의 단점
- 추가적인 저장 공간 사용
- 인덱스를 관리하기 위해 추가 작업이 필요

### 인덱스를 사용하기 적절한 경우
- INSERT, UPDATE, DELETE가 자주 발생하지 않는 컬럼
- JOIN이나 WHERE 또는 ORDER BY에 자주 사용되는 컬럼
- 데이터의 중복도가 낮은 컬럼

## PK
테이블 내에서 레코드의 위치를 유일하게 식별할 수 있는 칼럼 또는 그 조합.<br/>
``NOT NULL``, ``유니크`` 등의 특성을 갖는다.<br/>
레코드의 저장 위치를 결정한다.<br/>
PK가 유사한 레코드들끼리 묶어서 저장하기 때문에 __클러스터 인덱스__ 라고도 불린다.<br/>

### PK의 자료 구조
인덱스와 마찬가지로 트리 구조로 관리된다.<br/>
리프 노드가 아닌 노드는 PK와 PK가 속한 자식 노드를 저장.<br/>
리프 노드는 레코드의 메모리 주소를 저장<br/>

### 인덱스와 PK의 관계
인덱스는 PK에 의존하므로 인덱스를 통해 테이블에서 데이터를 가져오려면 2번의 작업이 필요<br/>

### 장점
- PK로 검색할 때 처리가 매우 빠름
- 연속되는 PK로 조회할 경우 랜덤 I/O가 아닌 순차 I/O를 사용하여 처리 속도가 더욱 빠름

### 단점
- 인덱스를 통해 검색할 때 PK로 다시 한번 검색해야 하므로 처리 성능이 느림
- INSERT 시에 PK에 의해 레코드의 저장 위치가 결정되기 때문에 처리 성능이 느림
- PK를 변경할 때 레코드를 DELETE 및 INSERT 해야 하므로 처리 성능이 느림

## Database Index
### Database Index 추가의 장단점은?
인덱스는 데이터의 검색 속도를 향상시키기 위해서 별도의 자료 구조를 사용합니다. 따라서 검색 속도가 향상됩니다. 하지만 인덱스를 저장하기 위해 추가적인 저장 공간을 사용하고, 인덱스를 관리하는 추가 작업이 필요합니다.<br/>

### 인덱스를 관리하는데 필요한 추가 작업이란?
MySQL의 InnoDB를 기준으로 설명하면, 인덱스는 B+트리 자료 구조를 사용합니다. 따라서 값이 정렬되는 순서를 고려하여 자료를 삽입, 삭제해야 합니다. 이 과정에서 인덱스의 트리를 재구성해야 할 수 있습니다.<br/>
